## 1. Public / Guest Сценарии

### 1.1 Просмотр списка событий

**Цель**: Guest может видеть доступные для бронирования события

**Шаги**:
1. Guest открывает главную страницу `/`
2. Система отображает список активных событий
3. Guest видит: название, описание, длительность каждого события

**Ожидаемый результат**:
- ✅ Отображаются только активные события
- ✅ Неактивные события скрыты
- ✅ Если нет событий — показывается empty state

**Edge cases**:
- ❌ Ошибка загрузки событий → показывается сообщение об ошибке

---

### 1.2 Выбор даты и времени бронирования

**Цель**: Guest может выбрать удобное время для встречи

**Шаги**:
1. Guest кликает "Записаться" на карточке события
2. Система открывает страницу календаря `/e/{slug}`
3. Guest выбирает дату в календаре
4. Система загружает доступные слоты для этой даты
5. Guest выбирает временной слот

**Ограничения**:
- ❌ Нельзя выбрать прошедшую дату
- ❌ Нельзя выбрать дату дальше 14 дней
- ❌ Нельзя выбрать занятый слот (отображается как disabled)
- ❌ Выходные дни без расписания — слоты не отображаются

**Ожидаемый результат**:
- ✅ Доступные даты кликабельны
- ✅ Недоступные даты disabled
- ✅ При выборе даты загружаются слоты
- ✅ Занятые слоты помечены как недоступные

---

### 1.3 Заполнение формы бронирования

**Цель**: Guest может оставить свои данные для бронирования

**Шаги**:
1. Guest выбрал слот, система перенаправляет на `/e/{slug}/book?slot={time}`
2. Guest заполняет форму:
   - Имя (обязательно, max 100 символов)
   - Email (обязательно, валидный формат)
   - Примечания (опционально)
3. Guest нажимает "Подтвердить запись"
4. Система создаёт бронирование

**Валидация**:
- ❌ Пустое имя → ошибка "Обязательное поле"
- ❌ Невалидный email → ошибка формата
- ❌ Имя > 100 символов → ошибка длины
- ❌ Слот уже занят (между выбором и отправкой) → ошибка конфликта

**Ожидаемый результат**:
- ✅ При валидных данных → редирект на страницу успеха
- ✅ Показывается подтверждение с деталями бронирования
- ✅ Генерируется cancel token для управления

---

### 1.4 Управление бронированием (отмена)

**Цель**: Guest может отменить своё бронирование

**Шаги**:
1. Guest получает ссылку управления `/bookings/{id}/success?token={token}`
2. На странице успеха отображается:
   - Подтверждение бронирования
   - Кнопка "Отменить запись"
   - Кнопка "Копировать ссылку управления"
3. Guest нажимает "Отменить"
4. Система показывает диалог подтверждения
5. Guest подтверждает
6. Бронирование отменяется

**Ограничения**:
- ❌ Неверный token → ошибка доступа
- ❌ Уже отменённое бронирование → соответствующее сообщение

**Ожидаемый результат**:
- ✅ Бронирование помечено как cancelled
- ✅ Показывается сообщение об успешной отмене
- ✅ Освобождается слот для других бронирований

---

### 1.5 Обработка ошибок (Guest)

| Ошибка | Поведение системы |
|--------|-------------------|
| Событие не найден | 404 страница |
| Слот уже занят | Сообщение "Это время уже забронировали" |
| API недоступен | Сообщение "Не удалось загрузить данные" |
| Невалидный токен | Сообщение "Неверная ссылка управления" |

## 2. Admin Сценарии

### 2.1 Управление событиями (CRUD)

#### Создание события

**Шаги**:
1. Admin открывает `/admin/events`
2. Нажимает "Новое событие"
3. Заполняет форму:
   - Название (обязательно)
   - Slug (обязательно, уникальный, формат: kebab-case)
   - Длительность в минутах (5-480)
   - Описание (опционально)
   - Активность (toggle)
4. Slug автогенерируется из названия (транслитерация для русского)
5. Сохраняет

**Валидация**:
- ❌ Дубликат slug → ошибка "Такой slug уже существует"
- ❌ Невалидный slug (начинается с цифры) → ошибка формата
- ❌ Длительность вне диапазона → ошибка

#### Редактирование события

**Шаги**:
1. Admin открывает список событий
2. Нажимает "Редактировать" на карточке
3. Может изменить: название, длительность, описание, активность
4. **Slug редактировать нельзя** (disabled поле)
5. Сохраняет изменения

#### Удаление события

**Шаги**:
1. Admin открывает редактирование события
2. Нажимает "Удалить"
3. Подтверждает в диалоге

**Ограничения**:
- ❌ Нельзя удалить событие с активными бронированиями
- ❌ Сообщение: "Нельзя удалить событие с активными бронированиями"

**Ожидаемый результат**:
- ✅ Admin видит ВСЕ события (включая неактивные)
- ✅ События отсортированы по дате создания (новые сверху)
- ✅ Изменения сохраняются и отражаются в публичной части

---

### 2.2 Управление расписанием

**Цель**: Admin настраивает доступные часы для бронирования

**Шаги**:
1. Admin открывает `/admin/schedule`
2. Настраивает для каждого дня недели:
   - Включён/выключен (enabled toggle)
   - Временные блоки (startTime, endTime)
3. Можно добавить несколько блоков в один день
4. Сохраняет расписание

**Валидация**:
- ❌ Пересекающиеся блоки → ошибка
- ❌ End < Start → ошибка
- ❌ Невалидный формат времени → ошибка

**Ожидаемый результат**:
- ✅ Расписание сохраняется
- ✅ Guest видит слоты только в настроенных блоках
- ✅ Выключенные дни не показывают слотов

---

### 2.3 Управление бронированиями

**Цель**: Admin видит и управляет всеми бронированиями

**Шаги**:
1. Admin открывает `/admin/bookings`
2. Видит список всех активных бронирований
3. Список отсортирован по времени начала (ближайшие сверху)
4. Для каждого бронирования видит:
   - Имя гостя
   - Email
   - Название события
   - Время начала
   - Статус
5. Может отменить бронирование (без токена)

**Фильтрация**:
- ✅ Показываются только активные бронирования
- ✅ Cancelled бронирования скрыты
- ✅ Прошедшие бронирования скрыты

**Безопасность**:
- ✅ Admin не видит cancelToken гостей
- ✅ Admin может отменить любое активное бронирование

## 3. API Сценарии

### 3.1 Публичное API

| Endpoint | Описание | Ограничения |
|----------|----------|-------------|
| `GET /api/v1/public/events` | Список активных событий | Только active=true |
| `GET /api/v1/public/events/{slug}` | Детали события | 404 если inactive |
| `GET /api/v1/public/events/{slug}/slots?date={YYYY-MM-DD}` | Доступные слоты | 400 если дата вне 14-дневного окна или в прошлом |
| `POST /api/v1/public/events/{slug}/bookings` | Создание бронирования | 409 если слот занят, 400 если невалидное время |
| `DELETE /api/v1/public/bookings/{id}?cancelToken={token}` | Отмена бронирования | 403 если неверный токен |

### 3.2 Admin API

| Endpoint | Описание | Особенности |
|----------|----------|-------------|
| `GET /api/v1/events` | Все события (включая inactive) | Полный список |
| `POST /api/v1/events` | Создание события | Валидация slug, дубликатов |
| `PATCH /api/v1/events/{slug}` | Обновление события | Slug неизменяем |
| `DELETE /api/v1/events/{slug}` | Удаление события | Блокировка если есть бронирования |
| `GET /api/v1/bookings` | Список бронирований | Без cancelToken, только active |
| `DELETE /api/v1/bookings/{id}` | Отмена бронирования | Admin может отменить любое |
| `GET /api/v1/schedule` | Получение расписания | Текущее расписание |
| `POST /api/v1/schedule` | Обновление расписания | Валидация блоков |

```
Admin                          System                    Guest

  |                              |                        |
```

**Проверки**:

```
Guest 1                        System                    Guest 2

  |                              |<── Открывает календарь |
```

**Проверки**:

- ✅ Сообщение об ошибке понятно пользователю

## 5. Структура тестовых данных

### События (Events)
```typescript
{
  title: string;        // "Intro Call", "Встреча на 30 минут"
  slug: string;         // "intro-call", "vstrecha-na-30-minut"
  duration: number;     // 15, 30, 45, 60, 90
  description?: string; // Опциональное описание
  active: boolean;      // Видно ли гостям
}
```

### Гости (Guests)
```typescript
{
  name: string;         // Имя фамилия
  email: string;        // valid@email.com
  notes?: string;       // Пожелания, вопросы
}
```

### Расписание (Schedule)
```typescript
{
  [day: string]: {
    enabled: boolean;   // Рабочий ли день
    blocks: Array<{
      startTime: string; // "09:00"
      endTime: string;  // "17:00"
    }>
  }
}
```

## 8. Troubleshooting

### Тест падает с "Page crashed"
- Увеличить таймаут в `playwright.config.ts`
- Проверить доступность фронтенда перед запуском
- Использовать `retries: 1` для нестабильных тестов

### "Timeout exceeded waiting for selector"
- Проверить что селектор актуален
- Увеличить `timeout` в `page.waitForSelector()`
- Добавить `await page.waitForLoadState('networkidle')`

### API моки не работают
- Проверить что MSW включён в `vite.config.ts`
- Убедиться что `handlers.ts` импортирован
- Проверить `baseURL` в playwright config

### Браузеры не установлены
```bash
npx playwright install chromium firefox webkit
```
